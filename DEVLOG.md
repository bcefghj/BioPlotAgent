## 第一章：调研——站在巨人的肩膀上

### 起点：一个朴素的需求

用户的需求很简单：
> "帮我开发一个小白也可以使用的生物信息学绘图 Agent，可以生成图表，又可以学，又可以用。"

这个需求看似简单，背后隐藏着一个真实的痛点——**生物信息学的绘图门槛实在太高了**。

一个生物医学的研究生，手里有 RNA-seq 数据，想画个火山图，需要：
1. 学 R 或 Python（2-3个月）
2. 安装复杂的环境（经常报错）
3. 学 ggplot2 或 matplotlib（又是几周）
4. 调颜色、调字体、调图例……

很多人还没画出图，就已经放弃了。

### 做了什么调研

在动手之前，我先搜索了业界已有的解决方案，了解：

**1. 学术前沿：PlotGDP（bioRxiv 2026）**

PlotGDP 是 2026 年初发表在 bioRxiv 上的一个基于 LLM 的生信绘图平台。它的核心思路是：**用预制的模板脚本降低 LLM 的幻觉风险**，让 AI 不是从零生成代码，而是在模板上做参数调整。

这个思路非常重要，直接影响了我们的架构选择。

**2. 工程实践：PlotAgent（PyPI）**

另一个叫 PlotAgent 的工具基于 LangGraph，用 Plotly 生成可视化，支持沙箱代码执行。它的亮点是**工具调用（tool-calling）**架构，AI 可以真正"执行"代码而不只是"生成"代码。

**3. 生态参考：微生信 / SRplot / Hiplot**

微生信（bioinformatics.com.cn）是国内最大的生信绘图在线平台之一，SRplot 声称有"超过 100 种可视化功能"。仔细研究它们支持的图表类型，形成了我们后来的图表分类体系：

```
差异表达分析 → 火山图、热图、MA Plot
降维聚类 → PCA、相关性热图
临床分析 → 生存曲线、森林图、ROC
富集分析 → GO柱状图、气泡图
基础统计 → 箱线图、小提琴图、散点图...
集合分析 → 韦恩图、UpSet Plot
基因组 → 曼哈顿图
关系流程 → 桑基图、瀑布图、雷达图
```

**4. AI 底座：MiniMax API**

用户提供了 MiniMax 的 API 套餐（MiniMax M2.7 模型）。
查阅文档后确认：MiniMax 支持 OpenAI 兼容接口，base_url 是 `https://api.minimax.io/v1`，可以直接用 `openai` Python SDK 调用，无需额外适配。

这让技术方案变得非常简洁。

## 第二章：架构设计——"能用"比"完美"更重要

### 核心设计决策

在正式写代码之前，做了几个关键的架构决策：

**决策 1：用 Streamlit，不用 FastAPI + React**

原因：目标用户是"零基础小白"，他们需要的是立刻能用，不是等一个部署好的服务。Streamlit 一行命令启动，适合个人工具和开源项目。

**决策 2：模板优先，AI 辅助，而不是纯 AI 生成**

受 PlotGDP 启发，每种图表都写死模板代码，参数化所有可调选项，AI 负责的是：
- 用自然语言解释图表
- 推荐合适的图表类型
- 回答用户问题

这样做的好处：**零幻觉风险**。火山图就是火山图，不会因为 LLM 一时"发挥"生成错误的图表代码。

**决策 3：三模式设计**

```
快速绘图模式 → 为"想出图"的用户
AI 助手模式  → 为"不知道画什么"的用户
学习中心模式 → 为"想学知识"的用户
```

这三种模式分别对应三类人群：赶 ddl 的研究生、刚入门的本科生、想提升的学者。

**决策 4：寓教于乐的教学设计**

每种图表不只是"会画"，而是配套：
- 通俗比喻解释概念（火山图 → "站在火山口往下看"）
- 数据格式说明
- 解读指南（横轴/纵轴/颜色代表什么）
- 动手练习（示例数据 + 实时调参）

### 遇到的第一个 bug：PCA 转置问题

在生成 PCA 展示图时遇到了一个经典的生信数据形态问题——

原始表达矩阵是"基因 × 样本"（200行基因，12列样本），但 PCA 需要"样本 × 基因"（12行样本，200列特征）。代码里的自动转置逻辑在加入分组信息（`group_col`）后出现了索引越界：

```python
# 错误代码（造成 IndexError）
sample_names = data.columns[df.columns.isin(numeric_df.index) | True]

# 修复后
sample_names = numeric_df.index.values
```

同时改进了转置判断逻辑：当用户已经提供 `group_col` 时，信任用户的数据行布局，不再强制转置。

这个 bug 提醒了一件事：**生物信息学数据的行列含义经常是反的**，这是很多初学者困惑的根源之一。

## 第六章：第二次迭代——从 10 种扩展到 25 种

### 为什么要扩展

> 用户："图的种类可以多一点，你可以看看微生信，看看其他网站，尽可能多的图的种类，满足各种各样的人的需求"

这个反馈非常合理。10 种图表只能满足 RNA-seq 分析的基本需求，但生物信息学的场景太多了：

- 做 GWAS 的需要**曼哈顿图**
- 做荟萃分析的需要**森林图**
- 做诊断模型的需要 **ROC 曲线**
- 做单细胞数据的需要**小提琴图**
- 做免疫浸润的需要**堆叠柱状图**
- 多组集合比较需要 **UpSet Plot**（韦恩图超过3组就很难看）
- 展示通路关系需要**桑基图**
- 展示肿瘤响应需要**瀑布图**

### 15 种新图表的设计原则

**原则 1：每种图都有清晰的生物学应用场景**

不是为了凑数，每一种图都对应一个真实的分析需求：

```python
"forest":    "多变量Cox回归结果展示"
"roc":       "诊断模型敏感度/特异度评估"
"manhattan": "GWAS 全基因组关联分析"
"sankey":    "通路基因表达变化流向"
"waterfall": "肿瘤治疗响应排序"
```

**原则 2：参数尽量自动推断，减少用户配置负担**

每个绘图函数都有智能默认值：
- 自动检测数值列/类别列
- 自动推荐合适的配色
- 自动决定是否需要排序/标准化

**原则 3：统一的代码风格**

所有 25 个绘图模块共享同一套工具函数（`utils.py`）：
- `style_axis()` — 统一坐标轴样式，去掉上方和右侧边框
- `add_watermark()` — 统一添加水印
- `save_figure()` — 统一保存为 PNG/PDF
- `BIO_PALETTE` — 统一配色方案（红蓝灰为主色调）

### 为什么选这些库

| 库 | 版本 | 选择理由 |

这种兼容性设计极大降低了接入成本，不需要学习任何新的 API 格式。

## 第九章：遇到的挑战与解决方案

### 挑战 1：中文字体渲染

`matplotlib` 默认不支持中文，图表标题会显示为方块。

解决方案：在 `utils.py` 中全局设置字体优先级：

```python
matplotlib.rcParams["font.sans-serif"] = [
    "Arial Unicode MS",  # macOS
    "SimHei",            # Windows
    "DejaVu Sans"        # Linux/fallback
]
matplotlib.rcParams["axes.unicode_minus"] = False
```

### 挑战 2：小红书图片中的 emoji 渲染

Pillow 的 `ImageFont` 不支持 emoji，`🌋` 这类字符会变成方框。

解决方案：接受这个限制，改用**彩色标签设计**（圆角矩形 + 文字）替代 emoji，保持视觉效果的同时绕过了字体限制。

### 挑战 3：数据量差异极大

有的图表处理 7869 个 SNP（曼哈顿图），有的只处理 8 个基因（柱状图）。
两者的渲染时间和内存占用差异巨大。

解决方案：
- 热图自动截取方差最大的前 N 个基因（默认 50）
- 曼哈顿图用向量化操作代替循环
- 相关性热图在样本数 >20 时自动关闭数值标注

### 挑战 4：UpSet Plot 从零实现

Python 有 `upsetplot` 库，但引入一个只用于一种图的库增加了依赖负担。

决定用 `matplotlib` 的原生 API 手动实现 UpSet Plot 的三个子图（交叉柱状图、点矩阵、集合大小柱图），完全零额外依赖。

### 挑战 5：桑基图的 Bezier 曲线

标准的桑基图需要绘制平滑的曲线连接节点。`matplotlib` 本身没有内置的桑基图，常用的 `plotly` 版本格式复杂。

方案：使用 `matplotlib.path.Path` 的三次贝塞尔曲线（`CURVE4`）手动绘制流量带，叠加透明度实现视觉分层效果。效果接近 `plotly` 的桑基图，但完全基于 matplotlib，保持风格统一。

## 第十一章：设计哲学

### 降低门槛，不是降低质量

BioPlotAgent 的核心设计原则是：**让操作简单，但不让结果廉价**。

每张图都是 300 DPI 的发表级输出，支持矢量 PDF。
火山图的基因标注有自动避免重叠算法。
生存曲线有 Log-rank 检验 p 值和置信区间。

这些细节，哪怕用户不知道，图表质量也是专业的。

### 从"工具"到"老师"

学习中心的设计理念不是简单的文档，而是：

```
什么是这种图？（用比喻解释）
       ↓
什么时候用？（实际场景）
       ↓
怎么解读？（每个视觉元素的含义）
       ↓
动手画一张！（示例数据 + 实时调参）
```

这个流程参考了游戏设计中的"新手引导"概念：不是给你一本说明书，而是带你玩一遍。

### 透明的 AI

BioPlotAgent 的 AI 助手有一条设计原则：**永远不假装自己能做到做不到的事**。

如果 API Key 没配置，直接显示配置方法，不假装 AI 在"思考"。
如果数据格式不对，给出具体的格式要求，不给模糊的错误提示。
如果某种图不适合用户的数据，AI 会直接说，并推荐更合适的替代方案。

## 后记：一点感想

生物信息学工具的世界里有一个奇怪的现象：

专业工具的开发者往往是最厉害的程序员，他们把工具做得功能强大，参数完善，但文档只有英文，安装需要 conda 环境，示例数据在 GitHub 的某个犄角旮旯……

结果就是：会用的人不需要这个工具，需要这个工具的人根本用不起来。

BioPlotAgent 想做的事情不复杂：**把专业工具做成每个人都能用的样子**。

不是通过降低功能，而是通过降低门槛。
不是通过简化结果，而是通过简化过程。

如果有一个生物医学的本科生，在这个工具的帮助下，第一次画出了自己的火山图，第一次看懂了一张热图——那这个项目的意义就达到了。
